## 1. System Classification

**System Type:** Code Automation + Structured Extraction (Hybrid)

**Description:**
An optional LLM enhancement layer for an HTML-to-React CLI tool. The system uses a rules engine (Phase 1-2) to mechanically split HTML into component trees; the LLM acts as a "proofreader" -- validating component boundaries, handling rules-uncategorizable tags, refining component names, and suggesting cleanup. The LLM is strictly non-blocking: it enhances output quality when available but the tool works perfectly without it.

**Critical Failure Modes:**
1. LLM introduces incorrect component boundaries that didn't exist in rules-only output
2. LLM suggests destructive changes (removing nodes, restructuring hierarchy) beyond scope
3. LLM failure causes pipeline to crash instead of graceful degradation
4. Token cost estimation is wildly inaccurate, causing budget surprises
5. LLM generates syntactically invalid TSX/JSX in its suggestions

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer Tooling (CLI for HTML-to-React conversion)

**User Population:** Frontend developers converting HTML source files (e.g., from a design handoff, legacy site, or template) into production-ready React component trees with CSS Modules.

**Stakes Level:** Low

**Output Consequence:** Generated code is written to disk and reviewed by the developer before any production use. The LLM pass is strictly advisory -- its output is a suggestion the developer can accept, modify, or discard. A bad suggestion does not ship code; it wastes developer time.

### What Domain Experts Evaluate Against

```
Dimension: Component naming quality
Good (expert accepts): Component name reflects its visual or functional role based on its content and context (e.g., ArticleCard, UserProfileHeader, NavigationMenu). Names follow React conventions (PascalCase). Names are specific enough to distinguish the component from siblings.
Bad (expert flags): Generic names like Div1, Wrapper3, Component2. Names that misrepresent content (calling something HeaderCard that contains a pricing table). Inconsistent casing.
Stakes: Medium
Source: React component design conventions -- poor names force developers to read implementation to understand component purpose, defeating the abstraction.
```

```
Dimension: Component boundary correctness
Good (expert accepts): Boundaries align with semantic HTML structure (section/article/main/aside as natural split points), repeated element patterns extracted into reusable components (e.g., card items in a list), and clear single-responsibility units.
Bad (expert flags): Arbitrary splits that create unnecessary component files for trivial markup. Failure to extract clearly repeated patterns. Merging of semantically distinct sections that should be separate. Over-splitting (one div per component) or under-splitting (entire page as one component).
Stakes: High
Source: Component design best practices (single responsibility, DRY, semantic structure) -- incorrect boundaries force refactoring later.
```

```
Dimension: Semantic HTML preservation
Good (expert accepts): Semantic elements (nav, article, section, aside, header, footer, main) are preserved as component roots and not flattened into generic divs. The component tree mirrors the document's semantic structure.
Bad (expert flags): Semantic HTML elements converted to divs. Semantic landmarks lost in the component tree. The rules engine output passed to LLM has already lost semantic signal.
Stakes: Medium
Source: Accessibility (a11y) and SEO requirements -- semantic HTML is the foundation of accessible React apps. Lost semantics require manual remediation.
```

```
Dimension: Cleanup suggestion actionability
Good (expert accepts): Suggestions are specific, actionable, and non-destructive. Each hint targets a concrete issue: a redundant wrapper div, an unclear class name (red-button -> primary-cta), dead CSS. Hints are phrased as observations with rationale, not imperative commands.
Bad (expert flags): Vague hints ("could be improved", "consider cleaning up"). Destructive suggestions (remove a node that breaks layout). Hints that contradict each other. Suggestions that would require significant structural rework outside the proofreader scope.
Stakes: Medium
Source: Practitioner UX -- developers ignore hints that require more effort to evaluate than the hint saves.
```

```
Dimension: Structural validity of LLM output
Good (expert accepts): LLM suggestions are syntactically valid TypeScript/TSX. Naming suggestions follow identifier rules. Boundary confirmations are internally consistent (no circular component imports, parent-child relationships respected).
Bad (expert flags): Suggestions contain invalid JSX syntax, TypeScript type errors, or break the component import tree (importing a component from itself). Parsed output fails Zod schema validation.
Stakes: Critical
Source: Developer trust -- if the proofreader produces output that does not parse cleanly, developers lose confidence in the tool and stop using it.
```

### Known Failure Modes in This Domain

**1. Naming regression:** LLM renames a rule-generated component name to something worse -- less descriptive, less idiomatic, or actively misleading. Example: rules engine outputs `PricingTierCard`, LLM "improves" it to `Card12`. This is worse than the original.

**2. Phantom boundary changes:** LLM suggests a component boundary modification (split or merge) that breaks the visual layout or creates an import cycle. Developer must detect and revert manually.

**3. Destructive cleanup suggestions:** LLM suggests removing a wrapper div that carries layout CSS, breaking the rendered output. Developer must identify which suggestion caused the regression.

**4. Schema-parseable but semantically wrong output:** LLM returns valid JSON matching the Zod schema but with reasoning that is incorrect or misaligned with the component's actual role in the page. Example: confirming a boundary at the wrong level, or recommending a name based on the wrong heuristic.

**5. Proofreader silence failure:** LLM declines to validate (returns `approved: false` with no actionable feedback) on inputs that are genuinely good, causing the developer to see no benefit from the LLM pass when one exists.

No user data is transmitted to LLM providers unless the user explicitly configures a remote provider; the tool does not handle PII, PHI, financial data, or any regulated content class.

## 2. Framework Decision

**Selected Framework:** Direct SDK (openai + @anthropic-ai/sdk)

**Version:** Pin latest stable versions -- openai (^4.x), @anthropic-ai/sdk (^0.40.x)

**Rationale:**
The project team has already committed to D-01: direct SDK calls with no abstraction layer. For this Code Automation + Extraction hybrid use case, direct SDK is optimal -- minimal abstraction overhead, full control over provider config, and baseURL enables Ollama without code changes. This is a TypeScript/Node.js CLI tool, so SDK availability (Python SDKs don't fit) constrains the choice to openai + @anthropic-ai/sdk.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Vercel AI SDK | Adds unnecessary abstraction layer; only needed for unified streaming/tool-calling/RSC support which this use case doesn't require |
| LangChain | Overkill -- this is a simple SDK call, not a multi-step chain or RAG pipeline |
| LlamaIndex | Python-first, not designed for TypeScript CLI tooling |

**Vendor Lock-In Accepted:** Partial -- Direct SDK per provider is inherently provider-specific, but the baseURL trick for Ollama + configurable provider selection minimizes lock-in. Users can switch providers via config without code changes.

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `response_format` / `output_config.format` | Instructs the model to output JSON matching a schema | Every structured extraction call -- required for type-safe parsing |
| `zodResponseFormat()` / `zodOutputFormat()` | Wraps a Zod schema into the SDK's structured output format | Converting your TypeScript type definitions into LLM output constraints |
| `.parse()` / `.parsed_output` | SDK method that calls the LLM and parses the response in one step | Single-shot structured extraction calls (non-streaming) |
| `.stream()` / `stream.finalMessage()` | Two-phase: start a stream, await the final parsed message | When you want streaming chunks + final structured result |
| `encoding_for_model()` from tiktoken | BPE encoder for exact token counting | Pre-call cost estimation, context truncation |
| `baseURL` constructor option | Override the default API endpoint | Ollama/local models, proxies, Azure OpenAI |

### Common Pitfalls

1. **Missing `max_tokens` causes timeout hangs (Anthropic)** -- The Anthropic SDK sets a default timeout based on `max_tokens`. If omitted, requests may hang. Always set `max_tokens` explicitly (1024-2048 is reasonable for this use case).

2. **`dangerouslyAllowBrowser: true` not needed in Node.js CLI** -- The openai SDK blocks browser usage by default. For CLI/Node.js, this is irrelevant. Do not set this flag in the CLI context -- it is only for client-side web apps.

3. **Ollama requires explicit `apiKey` when using `baseURL`** -- When pointing openai SDK at Ollama (e.g., `baseURL: 'http://localhost:11434/v1'`), you must still provide an `apiKey` (can be any string, Ollama ignores it): `new OpenAI({ baseURL: '...', apiKey: 'ollama' })`.

4. **OpenAI `response_format` only works with `gpt-4o-*` and `gpt-4o` models** -- Older models like `gpt-4-turbo` do not support structured outputs via `response_format`. Use `gpt-4o-mini` or `gpt-4o` for this feature. The API will return a `validation_error` if unsupported.

5. **Anthropic `thinking` parameter: `type: 'enabled'` is deprecated** -- The Anthropic SDK v0.97.x warns on `thinking.type: 'enabled'`. Use `thinking.type: 'adaptive'` or `thinking.type: 'disabled'` instead. Adaptive thinking is recommended for code tasks.

6. **Structured output parsing re-parses JSON even on API-level parse** -- When using `zodResponseFormat()` / `zodOutputFormat()`, the SDK first parses the raw API JSON response, then validates it with Zod. A `safeParse` failure means the LLM did not emit valid JSON matching the schema -- this is a prompt or model issue, not a network issue.

7. **`tiktoken` encoder must be freed** -- Every `encoding_for_model()` call returns an encoder that holds WASM resources. Call `enc.free()` after use to avoid memory leaks in long-running CLI processes.

8. **Retry logic: SDK retries 2xx->4xx are NOT retried** -- The openai SDK auto-retries on 429 (rate limit) and 5xx errors, but NOT on 400 (bad request), 401 (auth), or 422 (validation errors). Structured output schema mismatches return 422. Log and surface to user rather than retry blindly.

## 4. Implementation Guidance

**Model Configuration:**

| Provider | Model | Use Case | Key Params |
|---------|-------|----------|------------|
| OpenAI | `gpt-4o-mini` | Default for structured extraction | `temperature: 0.2`, `max_tokens: 1024` |
| OpenAI | `gpt-4o` | Higher quality when needed | `temperature: 0.2`, `max_tokens: 2048` |
| Anthropic | `claude-sonnet-4-7-202520101` | Higher reasoning quality | `max_tokens: 1024`, `thinking: { type: 'adaptive' }` |
| Anthropic | `claude-haiku-4-5-20251001` | Budget/edge cases | `max_tokens: 1024` |
| Ollama | Any local model (e.g., `llama3.2`) | Offline / self-hosted | Same params as OpenAI, via `baseURL` |

For this "proofreader" role (D-13 to D-17), `gpt-4o-mini` is the recommended default -- cost-effective ($0.15/1M input tokens), fast, and sufficient for component boundary validation. Escalate to `gpt-4o` or `claude-sonnet-4-7` only if validation accuracy is insufficient.

**Core Pattern:**

The LLM acts as a **validation layer** on rules-engine output (D-08, D-09). The pattern is:

```typescript
// src/pipeline/steps/llm-review.ts
import { encoding_for_model } from 'tiktoken';

const MAX_INPUT_TOKENS = 60_000; // Leave room for output within context window

export async function runLLMReview(
  rulesOutput: ComponentTree,
  config: LLMConfig,
): Promise<LLMReviewResult> {
  const inputJson = JSON.stringify(rulesOutput);

  // D-04: Display token estimate before call
  const enc = encoding_for_model('cl100k_base'); // shared encoder for gpt-4o-mini and claude
  const tokenCount = enc.encode(inputJson).length;
  enc.free();

  const costEstimate = (tokenCount / 1_000_000) * INPUT_COST_PER_1M;
  console.warn(`[llm] ~${tokenCount} tokens (~${
    costEstimate.toFixed(4)
  }) -- calling ${config.provider}/${config.model}`);

  // Build messages (system prompt + rules output as user content)
  const systemPrompt = buildSystemPrompt(); // D-13 to D-17 scope definition
  const userContent = buildUserContent(rulesOutput);

  try {
    const result = await callProvider(config, systemPrompt, userContent);
    return result;
  } catch (error) {
    // D-11: graceful degradation on LLM error
    console.error(`[llm] error: ${error.message}, falling back to rules-only`);
    return { approved: false, boundary_changes: [], naming_suggestions: [], cleanup_hints: [], _fallback: true };
  }
}
```

**Tool Use:**

No external tools required for this use case. If future phases add tool use:

- OpenAI: Use `zodFunction()` from `openai/helpers/zod` to define typed tools
- Anthropic: Use the built-in `tools` array in `messages.create()` with tool definitions

For now, this is a pure text-in / JSON-out structured extraction pattern.

**State Management:**

No persistent state. Each CLI invocation:
1. Loads config (`.h2uirc` via cosmiconfig)
2. Runs rules engine (Phase 1-3 output)
3. Optionally runs LLM review (this phase)
4. Outputs result and exits

For future multi-turn conversation with LLM (if `llm.mode: 'always'` adds dialog), consider a simple in-memory message history array, not a database.

**Context Window Strategy:**

This is a **batch extraction** pattern, not a conversational one. Each call is:

```
[system prompt] + [rules output as JSON] -> [structured review JSON]
```

Key limits:
- `gpt-4o-mini`: 128k context window
- `claude-sonnet-4-7`: 200k context window
- Typical HTML component tree: <10k tokens

For the foreseeable scope, truncation is not needed. If input approaches 50k+ tokens:
1. Truncate the component tree to top-N nodes + summary
2. Pass a summary of the rest as a separate message with `role: 'system'` note
3. Do not let the LLM see partial HTML -- it needs the full component tree for boundary decisions

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs -- independent of framework choice.

### Structured Outputs with Pydantic (Zod)

This use case is **strictly batch extraction**: rules engine output goes in, structured review JSON comes out. Structured outputs are mandatory, not optional.

**Zod schema for this use case:**
```typescript
// src/llm/structured/review.ts
import { z } from 'zod';

export const BoundaryChangeSchema = z.object({
  component_id: z.string(),
  action: z.enum(['confirm', 'reject', 'modify']),
  reason: z.string().max(200),
});

export const NamingSuggestionSchema = z.object({
  original: z.string(),
  suggested: z.string(),
  rationale: z.string().max(100),
});

export const ComponentReviewSchema = z.object({
  approved: z.boolean(),
  boundary_changes: z.array(BoundaryChangeSchema),
  naming_suggestions: z.array(NamingSuggestionSchema),
  cleanup_hints: z.array(z.string().max(150)),
});
```

**OpenAI integration -- `zodResponseFormat()`:**
```typescript
import { zodResponseFormat } from 'openai/helpers/zod';

const completion = await client.chat.completions.parse({

// Access typed result:
```

**Retry logic:**
```typescript
const MAX_RETRIES = 2;

async function callWithRetry<T>(
  fn: () => Promise<T>,
  label: string,
): Promise<T> {
  for (let attempt = 1; attempt <= MAX_RETRIES; attempt++) {
    try {
      return await fn();
    } catch (error) {
      if (attempt === MAX_RETRIES) throw error;
      // Only retry on network errors, not schema validation errors (422/400)
      if (error instanceof APIError && error.status >= 400 && error.status < 500) {
        throw error; // Don't retry client errors
      }
      console.warn(`[llm] ${label} attempt ${attempt} failed: ${error.message}, retrying...`);
    }
  }
  throw new Error('unreachable');
}
```

Schema validation errors (Zod `safeParse` failure) indicate the LLM did not follow the schema -- retrying will not help unless the prompt is also updated. Log the raw output and the validation error for debugging.

### Async-First Design

Both SDKs are **async-first**. All network I/O returns Promises.

**Common mistake: blocking on an async SDK in an async context:**
```typescript
// BAD -- blocking in an async function
async function bad() {
  const result = client.messages.create({ ... }); // Returns APIPromise, not awaited
  console.log(result.content); // undefined -- it's a pending promise
}

// GOOD
async function good() {
  const result = await client.messages.create({ ... });
  console.log(result.content); // actual content
}
```

**Stream vs. await:**
- Use `stream: true` + iterate chunks when you want to show progressive output to the user (UX)
- Use `.parse()` (non-streaming await) when you need the structured result before proceeding -- this is the correct pattern for pipeline integration where downstream steps depend on the LLM output

For this pipeline, the LLM result is consumed by a later pipeline step, so **non-streaming `.parse()` is correct**. Streaming is only needed if you want to show the user live token output.

```typescript
// Correct for pipeline integration (non-streaming):
const review = await client.messages.parse({ ... });
// review is available here, proceed to next step

// Streaming only if showing live output to user:
const stream = client.messages.stream({ ... });
for await (const event of stream) { /* ... */ }
```

### Prompt Engineering Discipline

**System prompt vs. user prompt separation:**

```typescript
// System prompt: defines scope, role, constraints (static)
function buildSystemPrompt(): string {
  return `You are a component proofreader for an HTML-to-React conversion tool.

SCOPE (D-13 to D-17):
- Confirm or reject component boundaries created by the rules engine
- Suggest better component names when the rules engine used generic names
- Identify dead code, redundant nesting, or unclear class names

OUT OF SCOPE (D-17):
- Do not restructure the component hierarchy
- Do not abstract repeated patterns
- Do not rewrite component code
- Do not remove nodes without explicit reason

OUTPUT FORMAT:
Return ONLY valid JSON matching the provided schema. No markdown, no explanation outside the JSON.`;
}

// User prompt: dynamic input from rules engine
function buildUserContent(rulesOutput: ComponentTree): string {
  return `Rules engine output:\n\`\`\`json\n${JSON.stringify(rulesOutput, null, 2)}\n\`\`\``;
}
```

**Few-shot guidance:** Not needed for this use case. The Zod schema + system prompt scope definition is sufficient. Adding few-shot examples would increase token cost with marginal accuracy benefit for the straightforward boundary validation task.

**`max_tokens` must be explicit:** Never leave `max_tokens` unbounded in production. For this use case, 1024 tokens is sufficient for the structured review output (typically <500 tokens). Setting too-low `max_tokens` (e.g., 256) risks truncated output that fails schema validation.

### Context Window Management

This is a **batch extraction** pattern, not a conversational one. Each call is a single request-response with no conversation history accumulation.

For this use case:
- Typical input: rules engine output (~1-5k tokens for a medium HTML file)
- Typical output: structured review (~200-800 tokens)
- Context window risk: **low** for current inputs

**If input grows large (>50k tokens):**
1. **Truncate** the component tree to top-N nodes + summary
2. Add a system prompt note: `"Note: The following is a partial component tree. Focus validation on the top-level structure."`
3. Do not rely on truncation for critical business logic -- ensure the rules engine produces a flat enough structure

**Do not accumulate conversation history** -- each LLM call is independent. The rules engine is the source of truth; the LLM is only a proofreader. No multi-turn conversation is needed or desired.

### Cost and Latency Budget

**Per-call cost estimate (display-only, D-04):**

```typescript
// Token-based cost estimation (display-only, D-04, D-05)
const enc = encoding_for_model('cl100k_base');
const inputTokens = enc.encode(inputJson).length;
enc.free();

const COST_PER_1M_INPUT = {
  'gpt-4o-mini': 0.15,
  'gpt-4o': 2.50,
  'claude-sonnet-4-7': 3.50,
  'claude-haiku-4-5': 0.80,
};

const inputCost = (inputTokens / 1_000_000) * COST_PER_1M_INPUT[model];
const outputCostEstimate = (1024 / 1_000_000) * COST_PER_1M_INPUT[model] * 0.5; // rough 50% compression
console.warn(`[llm] ~${inputTokens} input tokens, ~$${(inputCost + outputCostEstimate).toFixed(4)} estimated`);
```

**Caching:** D-06 explicitly says no caching. Each conversion triggers a fresh call. This is correct for a proofreader -- cached responses would be stale as the rules engine output changes per run.

**Cheaper model routing:**
- Use `gpt-4o-mini` as the default (D-01)
- Escalate to `gpt-4o` or `claude-sonnet-4-7` only if `gpt-4o-mini` produces invalid JSON or low-quality boundary decisions
- For Ollama: free, use as fallback when network unavailable

**Latency targets:**
- `gpt-4o-mini`: ~500-1500ms for typical input
- `claude-sonnet-4-7`: ~1000-3000ms
- Ollama (local): varies by hardware, typically 200-2000ms

For CLI UX, display a spinner during LLM call (reusing existing `ora()` spinner from Phase 3). Do not block on latency; async pipeline step handles this correctly.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| **Structural validity of LLM output** | **PASS:** LLM returns valid JSON that passes `ComponentReviewSchema.safeParse()`. All `naming_suggestions[].suggested` values are valid TypeScript identifiers (PascalCase, no reserved words). All `boundary_changes[].component_id` values reference IDs present in the input rules output. No circular import paths in boundary confirmations. | **Code:** Zod `safeParse` validation; regex check `^[A-Z][a-zA-Z0-9]*$` on suggested names; ID reference validation; cycle detection in boundary graph. | **Critical** |
| **Component boundary correctness** | **PASS:** LLM confirmations align with semantic HTML landmarks (section/article/main/aside as natural split points). Repeated patterns are confirmed for extraction. LLM does not confirm boundaries that merge semantically distinct sections. Over-splitting rate < 10% on known patterns. | **LLM Judge:** Calibrated evaluator (gpt-4o or claude-sonnet-4-7) scores boundary decisions against expert rubric. Requires calibration against 10 human-labeled examples (target >= 0.7 correlation before trusting). | **High** |
| **Component naming quality regression** | **PASS:** LLM-suggested names are not worse than rules-engine originals. `suggested` names are not generic (not matching `Div\d+`, `Wrapper\d+`, `Component\d+`). Names are PascalCase. Names reflect content (not `Card12` for a pricing table). | **LLM Judge:** Calibrated evaluator scores each naming suggestion pair (original vs. suggested) as improved / unchanged / regressed. Regression rate must be < 10% on reference dataset. | **Medium** |
| **Cleanup suggestion actionability** | **PASS:** Each cleanup hint is specific enough to act on without further clarification. Hints target concrete issues (redundant wrapper, unclear class name, dead CSS). No hint contradicts another. No hint suggests removing a node that carries layout CSS without explicit flagging. | **LLM Judge:** Evaluator scores each cleanup hint as actionable / vague / destructive. Actionable rate must be >= 80% on reference dataset. | **Medium** |
| **Semantic HTML preservation** | **PASS:** Semantic HTML elements (nav, article, section, aside, header, footer, main) appear as component roots in the final output. LLM does not confirm boundaries that flatten semantic landmarks into generic divs. | **Code:** Check component tree roots against known semantic tag list. Compare rules-only output to LLM-output for semantic tag loss. | **Medium** |
| **Graceful degradation on LLM failure** | **PASS:** When LLM call throws (network error, auth error, timeout), pipeline falls back to rules-only output within the same invocation. User sees `[llm] error: ... falling back to rules-only`. No crash. Exit code 0. | **Code:** Mock LLM failure in test harness. Assert fallback path executes. Assert `_fallback: true` flag in returned result. | **Critical** |
| **Token cost estimation accuracy** | **PASS:** Pre-call token estimate is within 20% of actual token count (measured via tiktoken). Cost estimate is within 30% of actual API cost. | **Code:** Compare `encoding_for_model('cl100k_base').encode(inputJson).length` to actual token count reported in API response. | **Medium** |

### Eval Tooling

**Primary Tool:** Promptfoo (CLI-first, no platform account required)

**Secondary:** Code-based validators (Jest/Vitest test suite for schema validation, regression detection, and deterministic checks)

**No tracing platform required:** This is a CLI tool with ephemeral invocations. Traditional tracing (Langfuse, Arize Phoenix) adds infrastructure overhead disproportionate to the eval needs. Structured JSON logging to local files is sufficient for post-run debugging.

**Setup:**
```bash
npm install --save-dev promptfoo@latest
```

Create `promptfooconfig.yaml` for structured output validation:
```yaml
providers:
  - id: openai:gpt-4o-mini
    label: gpt-4o-mini

prompts:
  - prompt: "system: {{system_prompt}}\n\nuser: {{input_json}}"
    variables:
      system_prompt: "You are a component proofreader for an HTML-to-React conversion tool. [full system prompt from buildSystemPrompt()]"
      input_json: "{{input_json}}"

tests:
  - description: "Component review schema validation"
    vars:
      input_json: "{{input_json}}"
    assert:
      - type: "javascript"
        code: |
          const result = JSON.parse(output);
          const schema = z.object({
            approved: z.boolean(),
            boundary_changes: z.array(z.object({
              component_id: z.string(),
              action: z.enum(['confirm', 'reject', 'modify']),
              reason: z.string(),
            })),
            naming_suggestions: z.array(z.object({
              original: z.string(),
              suggested: z.string(),
              rationale: z.string(),
            })),
            cleanup_hints: z.array(z.string()),
          });
          const parsed = schema.safeParse(result);
          if (!parsed.success) {
            throw new Error('Schema validation failed: ' + parsed.error.message);
          }
```

**CI/CD Integration:**
```bash

# Run unit tests for graceful degradation and schema validation

# Run full eval suite in CI
npx promptfoo eval --config promptfooconfig.yaml --input eval/dataset.csv && npx vitest run test/llm/
```

### Reference Dataset

**Size:** 20 HTML examples minimum, to be expanded from production failure modes.

**Composition:**

| Category | Count | Description |
|----------|-------|-------------|

| Repeated pattern pages | 4 | Pages with card grids, list items, product rows -- known SPL-04 cases |

**Labeling:**

Each example requires:
1. **Ground-truth component tree** -- manually authored by a senior frontend developer, marking correct boundaries

Labeling timeline: Start during implementation (Phase 4 execution), not after. First 10 examples labeled before first LLM integration test. Remaining 10 labeled during first eval run.

**Dataset format:** CSV with columns: `html_file,named_rules_output,json,expert_component_tree,json,expert_naming_scores,json,notes`

## 6. Guardrails

### Online (Real-Time)

These guardrails run on every LLM invocation. Each adds minimal latency (1-5ms for schema validation).

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Zod schema parse guardrail** | SDK returns raw output; `ComponentReviewSchema.safeParse(result)` fails | Never show raw LLM output to user. Log raw output + parse error to `.h2uirc` debug log path. Show user: `[llm] warning: response format unexpected, showing suggestions without LLM context`. Fall back to rules-only display. Do NOT crash. |
| **Destructive cleanup blocker** | Any `cleanup_hints` entry matches patterns: `remove.*div`, `delete.*wrapper`, `strip.*container` (case-insensitive regex) AND the hint does not contain `WARNING:` or `may break` | Prefix hint with `[review carefully]` tag. Never auto-apply destructive hints. Present to user as a question, not a suggestion. |
| **Naming regression detector** | `naming_suggestions[].suggested` matches generic pattern `^(Div\|Wrapper\|Component\|Card)\d+$` OR `suggested` length < `original` length AND original was non-generic | Reject the suggestion silently. Do not surface the regression suggestion. Log to structured debug output. |
| **LLM timeout guard** | LLM call exceeds 30 seconds | Cancel request, log timeout event, fall back to rules-only. Do not hang CLI. |

### Offline (Flywheel)

These metrics are collected and analyzed as a batch, feeding back into prompt and model improvements. They do not block user-facing behavior.

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **Schema parse failure rate** | Every LLM invocation -- log `parse_success: boolean` + `model` + `provider` to JSONL file | If > 5% failure rate on a specific model/provider: investigate prompt clarity, then switch to more reliable model. Alert if > 20%. |
| **Naming regression rate** | All naming suggestions across reference dataset runs | If regression rate > 10%: re-calibrate LLM judge; if confirmed, tighten system prompt with explicit anti-regression instructions. |
| **Boundary confirmation disagreement rate** | Compare LLM boundary decisions to expert labels in reference dataset | If disagreement > 25% on any boundary type: add few-shot example for that case to system prompt, or switch to higher-quality model. |
| **Cleanup hint actionable rate** | Human review sample of 20 cleanup hints per eval run | If actionable rate < 80%: revise cleanup hint instruction in system prompt to emphasize specificity requirements. |
| **Token estimate accuracy** | Every LLM invocation -- compare estimated vs. actual tokens | If average error > 30%: recalibrate token estimator using model-specific encoding, or switch to a dedicated token counting library. |

## 7. Production Monitoring

**Tracing Tool:** Not applicable (CLI ephemeral invocation model)

This is a developer CLI tool, not a persistent web service. Traditional tracing platforms (Langfuse, Arize Phoenix, LangSmith) are designed for services with high request volume and require infrastructure hosting that adds disproportionate overhead for this use case.

**Structured local logging** is used instead:
```typescript
// Every LLM invocation writes a structured log line to ~/.h2uirc/llm-log.jsonl
// Fields: timestamp, provider, model, input_tokens, output_tokens, parse_success, duration_ms, fallback
```

Users can enable verbose logging via `DEBUG=llm` env var to see full request/response traces.

**Key Metrics to Track:**

| Metric | Collection Method | Target |
|--------|------------------|--------|
| LLM parse success rate | Code: log `parse_success` flag per invocation | >= 95% on gpt-4o-mini |
| Graceful degradation success rate | Code: test suite mocking LLM failure | 100% (pipeline always falls back, never crashes) |
| Naming regression rate | LLM judge on reference dataset | < 10% |
| Boundary disagreement rate | LLM judge calibrated against expert labels | < 20% |
| Token estimate error | Code: compare estimated vs. actual token count | < 20% average error |
| LLM invocation cost per 100 invocations | Code: sum of `input_tokens * price_per_1M + output_tokens * price_per_1M` | Monitor for drift; no hard target |

**Alert Thresholds:**

Since this is a CLI tool without a persistent monitoring server, alerts are **not real-time**. They are detected via periodic eval runs:

| Alert Condition | Detection Method | Action |
|-----------------|------------------|--------|
| Parse success rate < 90% on a specific model | Run reference dataset, aggregate `parse_success` | Switch to fallback model; do not block user but document in `--version` output |
| Naming regression rate > 20% | Reference dataset eval run | Freeze LLM pass until prompt fixed; document in changelog |
| Graceful degradation test failure | CI/CD test suite | Block merge; pipeline regression |
| Token estimate error > 40% average | Reference dataset eval run | Fix token estimator before next release |

**Smart Sampling Strategy:**

Because there is no production traffic (CLI tool, user-initiated), evaluation is driven entirely by:
1. **Reference dataset runs** -- triggered on every release candidate (pre-merge)
2. **User-reported issues** -- filed as GitHub issues with reproduction steps; added to reference dataset
3. **Internal dogfooding** -- team members use the tool on real HTML files; flag regressions

**Signal-based sampling for human review:**
- All reference dataset examples with LLM boundary decisions that disagree with expert labels go to human review
- All naming suggestions where LLM judge scored "regressed" go to human review
- All schema parse failures go to human review (logged, not shown to user)

**Evaluation Cadence:**
- **Pre-merge:** Full reference dataset run (all 20+ examples, all metrics)
- **Pre-release:** Human review of all high-priority disagreements (boundary correctness, naming regression)
- **Quarterly:** Expand reference dataset with new failure mode examples from user-reported issues
