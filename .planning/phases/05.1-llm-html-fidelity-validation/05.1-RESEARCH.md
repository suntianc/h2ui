# Phase 05.1: LLM Fidelity + CLI Simplification - Research

**Researched:** 2026-05-22
**Domain:** LLM integration refactoring + CLI argument simplification
**Confidence:** HIGH

## Summary

Phase 05.1 merges the two existing LLM pipeline steps (`llmReviewStep` + `llmModifyStep`) into a single `llmFidelityStep` that validates generated components against original HTML before approving output. The CLI is simultaneously simplified: `--type tsx|jsx` replaces `--typescript`/`--no-typescript`, LLM is enabled by default, and `--llm off` disables it. The `--llm-provider`, `--llm-model`, `--llm-mode` flags are removed from CLI (moved to config file).

**Primary recommendation:** Create a unified `llmFidelityStep` that receives `ctx.html` + `ctx.components` + `ctx.componentTree`, performs fidelity validation in the LLM prompt, and returns both review metadata AND modified component code in a single schema. Merge the two Zod schemas into one `FidelityResult` schema.

## 1. LLM Fidelity Validation Approach

### Current State

`ctx.html` is available in the pipeline context (set by `parseStep` in `src/pipeline/steps/parse.ts`). Currently, neither `llmReviewStep` nor `llmModifyStep` use `ctx.html` for comparison.

### Recommended Approach: LLM Self-Reported Fidelity

**Decision:** Use LLM to self-report fidelity against original HTML, not AST serialization diff.

**Rationale:**

| Approach | Pros | Cons |
|----------|------|------|
| AST serialization + diff | Exact structural comparison | Complex to implement, noisy diffs (formatting, whitespace) |
| Serialization + string diff | Simple | Ignores semantic equivalence (e.g., `class` vs `className`) |
| **LLM self-reported fidelity** | **Handles semantic equivalence, no complex impl** | **Requires good prompt engineering** |
| Parse both, compare in code | Fast, deterministic | Loses text content mapping, complex to handle all edge cases |

LLM self-reported fidelity is the correct choice because:
1. The LLM already understands React JSX semantics (e.g., `class` -> `className`)
2. The LLM can compare text content semantically (HTML entity decoding, whitespace normalization)
3. The LLM can identify missing attributes or structural mismatches
4. Single LLM call for review + modify + fidelity keeps the schema simple

### What to Compare

The LLM prompt should instruct the model to verify:

1. **Structure:** All semantic HTML elements (header, nav, main, section, footer, etc.) are represented as components or inline JSX
2. **Attributes:** Key attributes preserved â€” `id`, `href`, `src`, `alt`, `aria-*`, `data-*`, `type` (for inputs/buttons)
3. **Text content:** All visible text content is present in the generated JSX
4. **CSS classes/styles:** `class` attributes mapped to `className`, inline `style` attributes preserved
5. **Repeated patterns:** Components extracted for repeated patterns match the original instances

### Implementation: Add `originalHtml` to LLM Input

```typescript
// In llmFidelityStep, pass ctx.html alongside componentTree and components
const userContent = buildFidelityUserContent(
  ctx.html,          // NEW: original HTML
  ctx.componentTree,
  ctx.components
);
```

The prompt instructs the LLM to validate fidelity and include a `fidelity_report` in the response.

## 2. Merging llmReviewStep + llmModifyStep into llmFidelityStep

### Current State

| Step | Input | Output |
|------|-------|--------|
| `llmReviewStep` | `componentTree` | `llmResult` (approved, boundary_changes, naming_suggestions, cleanup_hints) |
| `llmModifyStep` | `componentTree` + `components` | Modified `components[]` array |

Both steps are added to the pipeline sequentially when `llmConfig.mode !== 'off'`:

```typescript

  const { llmReviewStep } = await import('../../pipeline/steps/llm-review.js');
  const { llmModifyStep } = await import('../../pipeline/steps/llm-modify.js');
  pipeline.addStep(llmReviewStep);
  pipeline.addStep(llmModifyStep);

```

### Recommended Approach: Single Step with Combined Schema

**Decision:** Create `src/pipeline/steps/llm-fidelity.ts` that:
1. Receives `ctx.html` + `ctx.componentTree` + `ctx.components`
2. Calls LLM once with combined prompt (review + modify + fidelity)
3. Returns `FidelityResult` containing both review metadata AND modified components

### New Combined Schema

```typescript
// src/llm/structured/fidelity.ts
import { z } from 'zod';

export const BoundaryChangeSchema = z.object({
  component_id: z.string(),
  action: z.enum(['confirm', 'reject', 'modify']),
  reason: z.string().max(200),
});

export const NamingSuggestionSchema = z.object({
  original: z.string(),
  suggested: z.string(),
  rationale: z.string().max(100),
});

export const FidelityResultSchema = z.object({
  // Review fields
  approved: z.boolean(),
  boundary_changes: z.array(BoundaryChangeSchema),
  naming_suggestions: z.array(NamingSuggestionSchema),
  cleanup_hints: z.array(z.string().max(150)),

  // Modify fields
  components: z.array(z.object({
    name: z.string(),
    code: z.string(),
    rationale: z.string(),
  })),

  // Fidelity fields (NEW)
  fidelity_report: z.object({
    structure_match: z.boolean(),
    attribute_preservation: z.array(z.object({
      component: z.string(),
      missing_attributes: z.array(z.string()),
    })),
    text_content_match: z.boolean(),
    css_preservation: z.boolean(),
    fidelity_notes: z.array(z.string().max(200)),
  }),

  _fallback: z.boolean().nullable().optional(),
});

export type FidelityResult = z.infer<typeof FidelityResultSchema>;
```

### Commander.js Choice Syntax

.option('--llm <mode>', 'LLM mode: on or off', /^(on\|off)$/, 'on')

Wait â€” `--llm off` needs to be handled carefully. In commander.js, `--llm <value>` expects a value. When user runs `--llm off`, the value is `"off"`.

**Better approach using a custom parser:**

```typescript
.option('--llm [mode]', 'LLM mode: on or off (default: on)', (value) => {
  if (value === true) return 'on';  // --llm with no value means 'on'
  return value === 'off' ? 'off' : 'on';
}, 'on')
```

Actually, the cleaner pattern is:

```typescript
.option('--llm <mode>', 'LLM mode: on | off', (value) => {
  if (value !== 'off') return 'on';
  return 'off';
}, 'on')
```

Then in `convertCommand`:
```typescript
if (options.llm === 'off') {
  llmConfig = undefined;  // LLM disabled
}
```

## 5. Common Pitfalls

### Pitfall 1: LLM Fidelity Without Original HTML Context

**What goes wrong:** LLM generates components that look correct in isolation but don't match the original HTML structure.

**How to avoid:** Always pass `ctx.html` to the LLM in `buildFidelityUserContent()`. The prompt must include the original HTML as a code block for comparison.

### Pitfall 2: Breaking Change for CLI Users

**What goes wrong:** Removing `--llm-provider`, `--llm-model`, `--llm-mode` flags breaks existing scripts.

**How to avoid:** The old flags should be parsed but produce a deprecation warning. The config file approach means these settings were always configurable there.

**Migration path:**
1. Parse old flags with `hideOption()` (commander.js)
2. If used, print warning: `--llm-provider/--llm-model/--llm-mode are deprecated. Configure in .h2uirc instead.`
3. Continue to respect the values (backward compat) for one release cycle

### Pitfall 3: Mode 'always' vs 'auto' Semantics Lost

**What goes wrong:** Old `--llm` flag set `mode='always'`, which was more aggressive than the new default `mode='auto'`.

**How to avoid:** Document that `mode='auto'` now triggers on warnings (same as before), and users who want always-on LLM can set `llm.mode: 'always'` in config file.

### Pitfall 4: Deleting Old Files Before Migration Complete

**What goes wrong:** Deleting `llm-review.ts` and `llm-modify.ts` breaks any code that imports them directly.

**How to avoid:** Keep the old files until all references are updated. Mark them `@deprecated` in JSDoc, then delete in a follow-up cleanup commit.

### New Pipeline Step (llm-fidelity.ts)

```typescript
import type { PipelineStep, PipelineContext } from '../../types/pipeline.js';
import { runLLMFidelity } from '../../llm/llm-fidelity.js';

export const llmFidelityStep: PipelineStep = {
  name: 'llm-fidelity',

  async run(ctx: PipelineContext): Promise<PipelineContext> {
    const newCtx = { ...ctx, warnings: [...ctx.warnings], errors: [...ctx.errors] };
    const llmConfig = ctx.options.llm;

    if (!llmConfig || llmConfig.mode === 'off') {
      return newCtx;
    }

    // Auto mode: check for warnings
    if (llmConfig.mode === 'auto') {
      const hasRelevantWarnings = ctx.warnings.some(w =>
        w.includes('ambiguous') || w.includes('unknown-attribute')
      );
      if (!hasRelevantWarnings) {
        return newCtx;
      }
    }

    if (!ctx.componentTree || !ctx.components || !ctx.html) {
      newCtx.warnings.push('Missing required context for LLM fidelity check');
      return newCtx;
    }

    try {
      const result = await runLLMFidelity(ctx.html, ctx.componentTree, ctx.components, llmConfig);

      // Apply modified components
      const updatedComponents = [...ctx.components];
      for (const comp of result.components ?? []) {
        const validation = validateBeforeWrite(comp.code);
        if (!validation.valid) {
          newCtx.warnings.push(`LLM component "${comp.name}" skipped due to validation failure`);
          continue;
        }
        const idx = updatedComponents.findIndex(c => c.name === comp.name);
        if (idx !== -1) {
          updatedComponents[idx] = { ...updatedComponents[idx], code: comp.code };
        }
      }

      return {
        ...newCtx,
        components: updatedComponents,
        llmResult: result,
      };
    } catch (err: any) {
      console.warn(`[llm-fidelity] error: ${err.message}, falling back to rules-only`);
      return {
        ...newCtx,
        llmResult: { approved: false, _fallback: true } as PipelineContext['llmResult'],
        warnings: [...newCtx.warnings, `LLM fidelity check failed: ${err.message}`],
      };
    }
  },
};
```

## 8. Open Questions

1. **Should fidelity failures block output?**
   - What we know: Currently LLM approval (`llmResult.approved`) is not checked anywhere in the pipeline
   - What's unclear: Should `fidelity_result.structure_match === false` cause the pipeline to fail or just warn?
   - Recommendation: Add `--strict` interaction â€” in strict mode, fidelity failures should block output

2. **How to handle the `mode: 'always'` users?**
   - What we know: Old `--llm` flag set `mode='always'`
   - What's unclear: Is `mode: 'always'` semantically different from `mode: 'auto'` in practice?
   - Recommendation: Keep `mode: 'always'` in config schema, document it as "run LLM always (not just on warnings)"

3. **Backward compatibility for deprecated CLI flags?**
   - What we know: Users may have scripts using `--llm-provider`, `--llm-model`, `--llm-mode`
   - What's unclear: Should we accept and warn, or break immediately?
   - Recommendation: Accept with deprecation warning for one release, then remove
