## 1. System Classification

**System Type:** Autonomous Agent

**Description:**
Self-repairing autonomous agent that converts HTML to React/Vue components. Receives an `--agent` flag, declares its plan (PLAN), executes conversion (EXECUTE), verifies output fidelity (VERIFY), and repairs failures (REPAIR) up to 3 attempts. Uses function calling to expose tools (read_file, write_file, run_pipeline, run_llm, verify_output). A separate Verifier Agent calculates confidence scores. Token usage is tracked but not limited.

**Critical Failure Modes:**
1. Agent infinite loops — repair retry without progress, consuming excessive tokens
2. Silent corruption — agent produces syntactically valid but semantically wrong components
3. Tool misuse — agent calls wrong tool or wrong parameters, corrupting output files
4. Confidence score manipulation — agent inflates self-assessment to avoid admitting failure
5. History poisoning — agent repeatedly tries the same failed strategy due to faulty history tracking

## 1b. Domain Context

**Industry Vertical:** Developer Tooling (Frontend Engineering)

**User Population:** Frontend developers who receive HTML prototypes or design exports and need to convert them into production-ready React/Vue component code. They are professionals who value correctness, code quality, and time savings over manual refactoring.

**Stakes Level:** Medium

**Output Consequence:** Generated component code is integrated into production applications. Developers will read, maintain, and extend this code. Poor output quality causes developers to spend significant time debugging, refactoring, or rewriting instead of building features.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|----------------------|-------------------|--------|--------|
| Visual/structural fidelity | Generated component renders with identical layout, spacing, and visual hierarchy as the original HTML when given the same CSS | Component renders with broken layout, missing elements, or incorrect visual hierarchy | Critical | Core tool purpose |
| Component boundary correctness | Semantic HTML sections split at appropriate boundaries; child components are properly imported by parent | Components split at wrong boundaries causing broken imports, or unrelated elements merged into a single component | High | SPL-01 through SPL-06 |
| Style extraction completeness | All inline styles extracted to CSS modules with correct property names (camelCase for React), values preserved | Styles missing, incorrectly converted (e.g., string stays as string instead of object), or duplicated | High | CSS-01 through CSS-07 |
| Event handler conversion accuracy | HTML handlers (onclick, onchange) converted to React/Vue equivalents (onClick, @click); logic preserved | Handlers dropped, incorrectly converted, or logic lost | High | JSX-08, VUE-03 |
| TypeScript correctness | Correct types inferred from HTML attributes; props properly typed | Type errors, incorrect types, or casts that bypass checking | Medium | JSX-10 default output |

### Known Failure Modes in This Domain

1. **Repair-induced structural drift**: Agent repair attempts progressively diverge component structure from original HTML as each "fix" subtly changes the DOM tree.
2. **Style loss during repair**: When fixing structural issues, the agent drops or corrupts inline style attributes (e.g., `padding: 10px 20px` becomes `padding: "10px 20px"` as string).
3. **Self-healing non-issues**: Agent "repairs" something that was not actually broken (false positive verification failure), introducing new bugs.
4. **Component name collision after split**: After extracting repeated patterns, agent creates naming collisions or incorrect import paths when writing multiple files.

## 2. Framework Decision

**Selected Framework:** LangGraph

**Version:** Latest stable (@langgraph/core)

**Rationale:**
LangGraph is the right fit because this autonomous agent needs production-grade fault tolerance with checkpointing (history persistence to `.h2ui/agent-history.json`), a structured execution loop (PLAN -> EXECUTE -> VERIFY -> REPAIR), and full TypeScript parity. The separate Verifier Agent pattern maps cleanly to LangGraph's multi-node graph architecture. The project already uses both OpenAI and Anthropic SDKs, making LangGraph's model-agnostic approach ideal. LangGraph's time-travel debugging is invaluable for inspecting agent state transitions.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| OpenAI Agents SDK | Vendor lock-in — project uses both OpenAI and Anthropic |
| Claude Agent SDK | Claude-only — project needs model flexibility |
| LangChain | Python-first with less robust checkpointing than LangGraph |
| CrewAI | Python-only, limited TypeScript support |
| Google ADK | Gemini-only lock-in |

**Vendor Lock-In Accepted:** Partial — LangGraph is model-agnostic but adds a framework dependency. Accepted because the fault-tolerance benefits outweigh the dependency risk.

### Entry Point Pattern

The **Graph API** (recommended for PLAN -> EXECUTE -> VERIFY -> REPAIR loop):

```typescript
import { StateGraph } from "@langchain/langgraph";
import { tool } from "langchain/core/tools";
import { MemorySaver, SqliteSaver } from "@langchain/langgraph/checkpointing";
import { ChatAnthropic } from "@langchain/anthropic";
import { z } from "zod";

// 1. Define tools with Zod schemas
const readFileTool = tool(
  async ({ filePath }: { filePath: string }) => {
    const fs = await import("fs/promises");
    return await fs.readFile(filePath, "utf-8");
  },
  {
    name: "read_file",
    description: "Read contents of a file from the filesystem.",
    schema: z.object({
      filePath: z.string().describe("Absolute path to the file to read."),
    }),
  }
);

const writeFileTool = tool(
  async ({ filePath, content }: { filePath: string; content: string }) => {
    const fs = await import("fs/promises");
    await fs.writeFile(filePath, content, "utf-8");
    return `Written ${content.length} characters to ${filePath}`;
  },
  {
    name: "write_file",
    description: "Write content to a file on the filesystem.",
    schema: z.object({
      filePath: z.string().describe("Absolute path to the file."),
      content: z.string().describe("Content to write to the file."),
    }),
  }
);

// 2. Define state schema
interface AgentState {
  messages: Array<{ role: string; content: string }>;
  plan: string | null;
  execution_result: string | null;
  verification_result: string | null;
  repair_attempts: number;
  confidence_score: number | null;
  failed_strategies: string[];
}

// 3. Define node functions
function planNode(state: AgentState) {
  // LLM generates plan, returns updated state
  return { ...state, plan: "Generated plan here" };
}

function executeNode(state: AgentState) {
  // Execute conversion using tools
  return { ...state, execution_result: "Execution result" };
}

function verifyNode(state: AgentState) {
  // Verify output fidelity
  return { ...state, verification_result: "Pass/Fail" };
}

function repairNode(state: AgentState) {
  // Attempt repair with retry logic
  return { ...state, repair_attempts: state.repair_attempts + 1 };
}

// 4. Define conditional edge logic
function shouldRepair(state: AgentState): "repair" | "end" {
  if (state.verification_result === "Fail" && state.repair_attempts < 3) {
    return "repair";
  }
  return "end";
}

// 5. Build the graph
const workflow = new StateGraph<AgentState>({ schema: z.object({}) as any })
  .addNode("plan", planNode)
  .addNode("execute", executeNode)
  .addNode("verify", verifyNode)
  .addNode("repair", repairNode)
  .addEdge("__start__", "plan")
  .addEdge("plan", "execute")
  .addEdge("execute", "verify")
  .addConditionalEdges("verify", shouldRepair, {
    repair: "repair",
    end: "__end__",
  })
  .addEdge("repair", "execute"); // loop back to execute after repair

// 6. Compile with checkpointing for history persistence
const checkpointer = new MemorySaver(); // or SqliteSaver for file persistence
const app = workflow.compile({ checkpointer });

// 7. Invoke with thread_id for state persistence
const result = await app.invoke(
  { messages: [], plan: null, execution_result: null,
    verification_result: null, repair_attempts: 0,
    confidence_score: null, failed_strategies: [] },
  { configurable: { thread_id: "conversion-session-1" } }
);
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| **StateGraph** | Graph-based workflow builder with typed state | Define PLAN -> EXECUTE -> VERIFY -> REPAIR loop |
| **Nodes** | Pure functions that read/write state | Each phase (planNode, executeNode, verifyNode, repairNode) |
| **Edges** | Connections between nodes (START, END, or conditional) | Route flow between phases |
| **Checkpointers** | Persist agent state between invocations | History tracking in `.h2ui/agent-history.json` |
| **@tool decorator** | Binds Zod-validated functions as LLM-callable tools | Expose read_file, write_file, run_pipeline, verify_output |
| **createReactAgent** | Prebuilt ReAct loop (LLM decides tool -> execute -> repeat) | Alternative to manual node/edge graph |
| **MemoryStore** | Cross-thread persistent storage | Share context between main agent and verifier agent |

### Common Pitfalls

1. **Mutable state in nodes** — LangGraph state must be treated as immutable. Return new state objects, never mutate `state` directly. Mixing Python dicts and TypedDict causes reducer conflicts.

2. **Missing `__root__` for single-field state** — If your state has only one field and you try to use `state["key"]` directly instead of `state.__root__`, you will get `KeyError`. Use `Annotated[T, operator.add]` for list fields.

3. **Checkpointer thread_id mismatch** — If you invoke without the same `thread_id`, state is not restored. Use a stable `thread_id` per conversion session and persist it to `.h2ui/agent-history.json`.

4. **Infinite loops without cycle detection** — LangGraph does not auto-detect loops. Always use conditional edges with bounded retry counts (e.g., `repair_attempts < 3`) or you risk infinite loops consuming tokens.

5. **Async/await mismatch with checkpointer** — The Python `MemorySaver` is synchronous but PostgresSaver is async. Using synchronous checkpointer with async invoke causes silent state loss. Use `await app.ainvoke()` with async checkpointers.

**Core Pattern:**

The autonomous agent uses a **StateGraph with conditional edges** for the PLAN -> EXECUTE -> VERIFY -> REPAIR loop:

```typescript
import { StateGraph } from "@langchain/langgraph";
import { tool } from "langchain/core/tools";
import { MemorySaver, SqliteSaver } from "@langchain/langgraph/checkpointing";
import { ChatAnthropic } from "@langchain/anthropic";
import { z } from "zod";
import * as fs from "fs/promises";
import * as path from "path";

// --- Tool Definitions ---

const readFileTool = tool(
  async ({ filePath }: { filePath: string }) => {
    return await fs.readFile(filePath, "utf-8");
  },
  {
    name: "read_file",
    description: "Read a file from the filesystem.",
    schema: z.object({
      filePath: z.string().describe("Absolute path to the file."),
    }),
  }
);

const writeFileTool = tool(
  async ({ filePath, content }: { filePath: string; content: string }) => {
    await fs.writeFile(filePath, content);
    return `Written ${content.length} chars to ${filePath}`;
  },
  {
    name: "write_file",
    description: "Write content to a file.",
    schema: z.object({
      filePath: z.string().describe("Absolute path."),
      content: z.string().describe("Content to write."),
    }),
  }
);

const runPipelineTool = tool(
  async ({ inputPath, outputPath }: { inputPath: string; outputPath: string }) => {
    // Calls existing Pipeline class from src/pipeline/
    const { Pipeline } = await import("../../pipeline/index.js");
    const pipeline = new Pipeline({ inputPath, outputPath });
    await pipeline.run();
    return `Pipeline complete: ${inputPath} -> ${outputPath}`;
  },
  {
    name: "run_pipeline",
    description: "Run the HTML-to-component conversion pipeline.",
    schema: z.object({
      inputPath: z.string(),
      outputPath: z.string(),
    }),
  }
);

const verifyOutputTool = tool(
  async ({ htmlPath, componentPath }: { htmlPath: string; componentPath: string }) => {
    // Returns structured verification result
    return JSON.stringify({ match: true, confidence: 0.92 });
  },
  {
    name: "verify_output",
    description: "Verify component output matches source HTML.",
    schema: z.object({
      htmlPath: z.string(),
      componentPath: z.string(),
    }),
  }
);

// --- State Schema ---

interface AgentState {
  messages: Array<{ role: "user" | "assistant" | "tool"; content: string }>;
  plan: string | null;
  current_phase: "PLAN" | "EXECUTE" | "VERIFY" | "REPAIR" | "DONE";
  input_path: string | null;
  output_path: string | null;
  repair_attempts: number;
  max_repair_attempts: number;
  verification_result: { match: boolean; confidence: number } | null;
  failed_strategies: string[];
  confidence_score: number | null;
}

// --- Node Implementations ---

function planNode(state: AgentState, config: { llm: ChatAnthropic }) {
  const systemPrompt = `You are an HTML-to-component conversion planner.
Analyze the input HTML file and declare your conversion plan.
Output your plan as a structured description of:
1. Component structure to generate
2. Framework target (React/Vue)
3. Key conversion decisions`;

  const response = config.llm.invoke([
    { role: "system", content: systemPrompt },
    ...state.messages,
  ]);

  return {
    ...state,
    plan: response.content as string,
    current_phase: "EXECUTE",
  };
}

function executeNode(state: AgentState, config: { tools: any[]; llm: ChatAnthropic }) {
  // Use LLM to decide tool calls based on plan
  const toolsByName = Object.fromEntries(config.tools.map((t) => [t.name, t]));

  const response = config.llm.bind_tools(config.tools).invoke([
    { role: "system", content: `Execute the plan: ${state.plan}` },
    ...state.messages,
  ]);

  // Execute any tool calls in the response
  const toolMessages = [];
  if (response.tool_calls) {
    for (const call of response.tool_calls) {
      const tool = toolsByName[call.name];
      const result = await tool.invoke(call.args);
      toolMessages.push({
        role: "tool" as const,
        content: result,
        tool_call_id: call.id,
        name: call.name,
      });
    }
  }

  return {
    ...state,
    messages: [...state.messages, response, ...toolMessages],
    current_phase: "VERIFY",
  };
}

function verifyNode(state: AgentState, config: { verifier_llm: ChatAnthropic }) {
  // Call the separate Verifier Agent for confidence scoring
  const verifyResponse = config.verifier_llm.invoke([
    {
      role: "system",
      content: `Score the fidelity of ${state.output_path} against ${state.input_path}.
Return JSON: { "match": boolean, "confidence": 0.0-1.0, "issues": string[] }`,
    },
  ]);

  const result = JSON.parse(verifyResponse.content as string);
  return {
    ...state,
    verification_result: result,
    current_phase: result.match ? "DONE" : "REPAIR",
  };
}

function repairNode(state: AgentState, config: { llm: ChatAnthropic }) {
  if (state.repair_attempts >= state.max_repair_attempts) {
    return { ...state, current_phase: "DONE" };
  }

  const strategy = getNextStrategy(state.failed_strategies);
  const repairPrompt = `Repair the conversion failure.
Previous strategies that failed: ${state.failed_strategies.join(", ")}
Try this strategy: ${strategy}
${state.plan}`;

  return {
    ...state,
    plan: repairPrompt,
    current_phase: "EXECUTE",
    repair_attempts: state.repair_attempts + 1,
    failed_strategies: [...state.failed_strategies, strategy],
  };
}

// --- Conditional Edge Logic ---

function routePhase(state: AgentState): string {
  switch (state.current_phase) {
    case "PLAN": return "execute";
    case "EXECUTE": return "verify";
    case "VERIFY": return state.verification_result?.match ? "__end__" : "repair";
    case "REPAIR": return state.repair_attempts >= state.max_repair_attempts ? "__end__" : "execute";
    case "DONE": return "__end__";
    default: return "__end__";
  }
}

// --- Graph Construction ---

const workflow = new StateGraph<AgentState>({
  channels: {
    // Define channels explicitly for AgentState
  },
})
  .addNode("plan", (state, config) => planNode(state, { llm: claudeModel }))
  .addNode("execute", (state, config) => executeNode(state, { tools: [readFileTool, writeFileTool, runPipelineTool, verifyOutputTool], llm: claudeModel }))
  .addNode("verify", (state, config) => verifyNode(state, { verifier_llm: claudeModel }))
  .addNode("repair", (state, config) => repairNode(state, { llm: claudeModel }))
  .addEdge("__start__", "plan")
  .addConditionalEdges("plan", (s) => "execute")
  .addConditionalEdges("execute", (s) => "verify")
  .addConditionalEdges("verify", (s) => s.verification_result?.match ? "__end__" : "repair")
  .addConditionalEdges("repair", (s) => s.repair_attempts >= s.max_repair_attempts ? "__end__" : "execute");

// --- Checkpointing for History Persistence ---

const checkpointer = new SqliteSaver({
  connectionPath: ".h2ui/agent-history.db", // Persists to file
});

const app = workflow.compile({ checkpointer });

// --- Invocation ---

async function runAgent(inputPath: string, outputPath: string) {
  const initialState: AgentState = {
    messages: [{ role: "user", content: `Convert ${inputPath} to component at ${outputPath}` }],
    plan: null,
    current_phase: "PLAN",
    input_path: inputPath,
    output_path: outputPath,
    repair_attempts: 0,
    max_repair_attempts: 3,
    verification_result: null,
    failed_strategies: [],
    confidence_score: null,
  };

  const config = {
    configurable: {
      thread_id: `conversion-${Date.now()}`,
    },
  };

  const finalState = await app.invoke(initialState, config);

  return {
    outputPath: finalState.output_path,
    confidence: finalState.confidence_score,
    repairAttempts: finalState.repair_attempts,
  };
}
```

**Tool Use Configuration:**

Tools are bound to the LLM via `bind_tools()`. Each tool uses Zod for schema validation:

- `read_file(filePath: string)` — reads file contents
- `write_file(filePath: string, content: string)` — writes file
- `run_pipeline(inputPath: string, outputPath: string)` — runs conversion pipeline
- `verify_output(htmlPath: string, componentPath: string)` — structured diff verification

**State Management Approach:**

1. **Thread-scoped checkpointing** via `SqliteSaver` at `.h2ui/agent-history.db`
2. **Thread ID** is `conversion-{timestamp}` — persisted per session for replay/debugging
3. **Failed strategies list** in state prevents repeating same repair approaches
4. **History file** at `.h2ui/agent-history.json` stores structured failure log for post-run analysis

**Context Window Strategy:**

For HTML files that may be large:
1. **First pass**: LLM sees only file path and metadata (size, line count)
2. **Chunked reading**: For files > 200KB, read in chunks and summarize
3. **Streaming**: Use LangGraph's `stream()` API for token-by-token output visibility
4. **Compaction**: PLAN phase produces a concise plan that fits in context; full HTML only in EXECUTE tools

### 4b.1 Structured Outputs with Pydantic

LangGraph's LLM nodes should use **Zod schemas** (TypeScript) or **Pydantic models** (Python) for structured tool inputs. For structured LLM outputs (e.g., verification results, confidence scores), use model binding with output parsers.

**TypeScript + Zod (recommended for this project):**

```typescript
import { z } from "zod";

// Tool input schema - validated before tool execution
const VerifyOutputSchema = z.object({
  htmlPath: z.string().describe("Path to source HTML"),
  componentPath: z.string().describe("Path to generated component"),
});

// Verification result schema - LLM output parsing
const VerificationResultSchema = z.object({
  match: z.boolean().describe("Whether output matches source"),
  confidence: z.number().min(0).max(1).describe("Confidence score 0-1"),
  issues: z.array(z.string()).optional().describe("List of fidelity issues"),
  repairedWith: z.string().optional().describe("Strategy used to repair"),
});

// Tool definition with Zod schema
const verifyOutputTool = tool(
  async ({ htmlPath, componentPath }: { htmlPath: string; componentPath: string }) => {
    // Tool implementation
    return JSON.stringify({ match: true, confidence: 0.92, issues: [] });
  },
  {
    name: "verify_output",
    description: "Verify component output matches source HTML.",
    schema: VerifyOutputSchema,
  }
);

// Using with_structured_output for verification results
import { withStructuredOutput } from "langchain/core/output_parsers";

const structuredVerifier = claudeModel.withStructuredOutput(VerificationResultSchema);

const result = await structuredVerifier.invoke([
  { role: "system", content: "Score fidelity of component vs HTML" },
  { role: "user", content: `HTML: ${htmlContent}
Component: ${componentContent}` },
]);

// result is typed as { match: boolean; confidence: number; issues?: string[] }
```

**Python + Pydantic:**

```python
from pydantic import BaseModel, Field
from langchain_core.utils.structured_output import StructuredOutputParser

class VerificationResult(BaseModel):
    match: bool = Field(description="Whether output matches source")
    confidence: float = Field(0.0, ge=0.0, le=1.0, description="Confidence score 0-1")
    issues: list[str] | None = Field(default=None, description="Fidelity issues")

output_parser = StructuredOutputParser.from_llm(
    schema=VerificationResult,
    llm=claudeModel
)

result = await output_parser.aparse(response.content)
```

**Retry Logic:**
- On schema validation failure: retry up to 2 times with shorter prompt
- Log failures to `.h2ui/agent-history.json` with `{ attempt, error, prompt }`
- Surface to user after 3 failures: "Verification parsing failed after 3 attempts"

### 4b.2 Async-First Design

LangGraph supports both sync (`invoke`) and async (`ainvoke`, `astream`) execution. For production, prefer async for I/O-bound tool calls.

**TypeScript async pattern:**

```typescript
// Correct: use ainvoke for async checkpointers
const result = await app.ainvoke(initialState, config);

// Stream tokens for UX (PLAN phase output to console)
for await (const chunk of app.astream(initialState, config)) {
  process.stdout.write(chunk);
}
```

**Common mistake: sync invoke with async checkpointer**

```typescript
// WRONG - causes silent state loss with PostgresSaver or async stores
const app = workflow.compile({ checkpointer: new PostgresSaver(pool) });
const result = app.invoke(initialState, config); // BUG: sync call with async checkpointer

// CORRECT
const result = await app.ainvoke(initialState, config); // async call
```

**Stream vs. await:**
- **Stream (`astream`)**: Use for PLAN phase output to stdout so user sees reasoning
- **Await (`ainvoke`)**: Use for EXECUTE/VERIFY/REPAIR when you need structured state at end

### 4b.3 Prompt Engineering Discipline

**System vs. user prompt separation:**

```typescript
// System prompt: defines agent role, rules, constraints
const SYSTEM_PROMPT = `You are an HTML-to-component conversion agent.
You must:
1. Declare a plan before executing
2. Use tools to read input and write output
3. Call verify_output after execution
4. Attempt repair strategies when verification fails
Never skip the PLAN phase.`;

// User prompt: task-specific context
const userMessage = { role: "user" as const, content: `Convert ${inputPath}` };

// LLM call: system first, then conversation history
llm.invoke([{ role: "system", content: SYSTEM_PROMPT }, ...state.messages]);
```

**Few-shot guidance:**
- For repair strategies: inline examples of successful repairs in the repair prompt
- For verification: provide 1-2 examples of {html, component, score} in the verifier system prompt
- For plan format: show example plan structure in the planner system prompt

**max_tokens budget:**
- Always set `max_tokens` explicitly — never leave unbounded in production
- PLAN phase: 1024 tokens (concise plan)
- EXECUTE phase: 4096 tokens (component generation)
- VERIFY phase: 512 tokens (structured result)
- REPAIR phase: 2048 tokens (strategy explanation)

### 4b.4 Context Window Management

**For the autonomous agent system:**

1. **PLAN phase**: Compact — LLM sees only file path and metadata, outputs concise plan
2. **EXECUTE phase**: Full — tools have direct file access; LLM sees tool results
3. **VERIFY phase**: Chunked — large HTML files are summarized before verification
4. **REPAIR phase**: Focused — only the failing portion of HTML and component

**Compaction pattern (LangGraph's built-in):**

```typescript
// After each phase, summarize the state to keep context manageable
function compactState(state: AgentState): AgentState {
  return {
    ...state,
    messages: state.messages.slice(-10), // Keep last 10 messages only
    plan: state.plan ? `Plan: ${state.plan.substring(0, 200)}...` : null,
  };
}
```

**Token tracking:**

```typescript
let totalTokens = 0;
function trackTokens(usage: { prompt_tokens: number; completion_tokens: number }) {
  totalTokens += usage.prompt_tokens + usage.completion_tokens;
  console.log(`Tokens used: ${totalTokens} / session`);
}
```

**Per-call cost estimates (Claude Sonnet 4 / GPT-4o):**

| Phase | Model | Input Tokens | Output Tokens | Est. Cost |

| PLAN | claude-sonnet-4-6 | ~500 | ~200 | $0.003 |
| EXECUTE | claude-sonnet-4-6 | ~2000 | ~3000 | $0.012 |
| VERIFY | claude-haiku-4 | ~1500 | ~100 | $0.001 |
| REPAIR | claude-sonnet-4-6 | ~1000 | ~500 | $0.005 |

| **Total (3 repairs)** | | ~7000 | ~4800 | **~$0.031** |

**Caching strategy:**

- **Exact-match cache**: Cache LLM responses by input hash for identical HTML files
- **Semantic cache**: For similar HTML (structural similarity > 0.9), reuse conversion
- **Tool results cache**: read_file results cached for same file path within session

**Sub-task model routing:**


**Latency targets:**
- PLAN: < 2s
- EXECUTE: < 10s per component
- VERIFY: < 1s (structured diff)
- REPAIR: < 5s
- **Total target (no repair): < 15s per file**

## 5. Evaluation Strategy

### Dimensions

All dimensions are grounded in the domain rubric ingredients from Section 1b. Each rubric uses domain expert language from the "What Domain Experts Evaluate Against" table.

| Dimension | Rubric | Measurement Approach | Priority |
|-----------|--------|---------------------|----------|
| **Visual/structural fidelity** | PASS: Generated component renders with identical layout, spacing, and visual hierarchy as the original HTML when given the same CSS. DOM structure preserved (semantic tags in place, nesting depth within tolerance).<br><br>FAIL: Component renders with broken layout, missing elements, incorrect visual hierarchy, or structural divergence after repair attempts. | Code (structural diff: DOM tag sequence, attribute count, nesting depth) + LLM Judge (semantic equivalence for complex layouts) | Critical |
| **Task completion** | PASS: Agent successfully converts HTML to component(s) with confidence >= 70% within max_repair_attempts=3, and outputs valid framework code (React .tsx or Vue .vue).<br><br>FAIL: Agent fails to produce valid output after 3 repair attempts, or reports confidence < 70% and still terminates, or produces syntactically invalid code that fails compilation. | Code (file existence, tsc/vue-tsc compilation pass, import graph valid) + LLM Judge (confidence calibration check) | Critical |
| **Tool use correctness** | PASS: Agent calls the correct tool (read_file/write_file/run_pipeline/verify_output) with correct parameters for each phase. Tool call sequence follows PLAN->EXECUTE->VERIFY->REPAIR contract.<br><br>FAIL: Agent calls wrong tool, passes incorrect file paths that would corrupt output, or skips required phases. | Code (tool call trace validation from LangSmith/LangGraph checkpoint) | Critical |
| **Loop detection / repair progress** | PASS: Agent makes measurable progress across repair attempts. Verification confidence increases OR verification issues decrease with each repair. Agent does not repeat the same failed strategy.<br><br>FAIL: Agent cycles through repair attempts without progress (confidence stays flat or decreases, same issues persist). Agent attempts the same strategy twice. Agent exceeds max_repair_attempts. | Code (confidence trend analysis across repair attempts, failed_strategies deduplication check) | Critical |
| **Component boundary correctness** | PASS: Semantic HTML sections (<header>, <nav>, <main>, <section>, <article>, <footer>) split at appropriate boundaries. Child components properly imported by parent via relative paths. Repeated patterns extracted as reusable components.<br><br>FAIL: Components split at wrong boundaries causing broken imports, import paths incorrect (wrong relative path, missing extension), or unrelated elements merged into a single component. | Code (import graph validation: all imported files exist, relative paths correct) + LLM Judge (boundary decision quality) | High |
| **Style extraction completeness** | PASS: All inline `style` attributes extracted to CSS modules (`.module.css` or `<style scoped>`) with correct property names (camelCase for React, kebab-case preserved for Vue), values preserved exactly, no string values left as raw strings.<br><br>FAIL: Styles missing from output, incorrectly converted (e.g., `padding: "10px 20px"` string instead of object), duplicated across components, or shorthand properties not expanded. | Code (regex extraction of style attributes from HTML vs. CSS modules in output, property name format validation) | High |
| **Event handler conversion accuracy** | PASS: HTML event handlers (onclick, onchange, oninput, onblur, etc.) converted to React equivalents (onClick, onChange, onInput, onBlur) or Vue equivalents (@click, @change, etc.). Handler logic (attribute references, simple expressions) preserved.<br><br>FAIL: Handlers dropped entirely, incorrectly converted (wrong prop name, wrong syntax), or logic lost (e.g., `onclick="doSomething()"` becomes `<button onClick>` with no handler). | Code (pattern matching: HTML handler attributes in source vs. React/Vue handlers in output) | High |
| **Confidence score accuracy** | PASS: Agent-reported confidence score correlates with actual verification outcome. If agent reports 90% confidence, independent verification agrees within 10%. No systematic inflation.<br><br>FAIL: Agent reports high confidence (>= 80%) but verification finds significant fidelity issues. Agent systematically under-reports failure likelihood. | LLM Judge (calibrated against independent verification) + Code (correlation analysis between agent confidence and verification score) | High |
| **History tracking correctness** | PASS: Agent reads `.h2ui/agent-history.json` before selecting repair strategy. Agent never attempts a strategy in the failed_strategies list. History file correctly persists across sessions.<br><br>FAIL: Agent repeats a strategy that is already in the failed_strategies list. History file corrupted or not written. Agent does not read history before selecting strategy. | Code (failed_strategies deduplication check, history file read/write validation) | High |
| **TypeScript correctness** | PASS: TypeScript compilation succeeds (tsc --noEmit passes). Props interface correctly typed based on HTML attributes. No `any` casts that bypass type checking except where explicitly allowed.<br><br>FAIL: TypeScript compilation errors. Props interface missing or incorrect. Excessive use of `any` casts. Wrong types inferred (e.g., `string` instead of `number` for `maxlength`). | Code (tsc --noEmit validation) | Medium |

## 6. Guardrails

### Online (Real-Time)

These guardrails run on every agent invocation, before output is returned to user. Each adds latency; keep them minimal.

| Guardrail | Trigger Condition | Intervention | Latency Impact |
|-----------|-------------------|--------------|----------------|
| **Max repair attempts exceeded** | `repair_attempts >= max_repair_attempts` (3) AND verification not passing | Block. Return last output with confidence score and warning: "Conversion completed with low confidence. Manual review recommended." | None (already in execution path) |
| **Token budget exceeded** | Cumulative token count > configured limit (default: 50,000 tokens) | Block. Terminate agent, return partial output if any, log to `.h2ui/agent-history.json`. | None (tracked inline) |
| **Tool call parameter validation** | Zod schema validation fails on tool call arguments | Block tool execution, return error to agent LLM, agent must re-plan | <10ms |
| **File write path traversal check** | `write_file` target path escapes output directory | Block. Log attempt to `.h2ui/agent-history.json` as potential misuse. Return error. | <5ms |
| **Silent corruption detection** | Agent reports confidence >= 80% but verification result shows confidence < 50% | Flag for offline review. Do not block, but add metadata flag to output. Surface warning to user. | <50ms (extra verification call) |

### Offline (Flywheel)

These metrics are collected continuously but evaluated as batch analysis, feeding the improvement loop. They do not block individual requests.

| Metric | Collection Method | Sampling Strategy | Action on Degradation |
|--------|------------------|-------------------|----------------------|
| **Confidence score drift** | Compare agent-reported confidence vs. independent verification confidence per run | 100% of runs with confidence < 80%; 10% of runs with confidence >= 80% | If mean delta > 15%, retrain verifier prompt or adjust calibration |
| **Repair loop frequency** | Count runs that exhaust all 3 repair attempts | 100% of exhausted-repair runs | If rate > 20% of all runs, analyze failure patterns, add new repair strategies to pool |
| **Style loss rate** | Detect missing or corrupted styles in output via CSS extraction validation | 20% random sample of all runs | If rate > 10%, add style-preservation constraint to REPAIR prompt |
| **Structural drift rate** | Detect progressive DOM divergence across repair attempts | 100% of multi-repair runs | If rate > 15%, add constraint to prevent structural modifications that diverge from original |
| **Confidence inflation index** | Ratio: agent-reported-high-and-verification-low / all agent-reported-high | 100% of runs where agent reports >= 80% | If inflation index > 0.1, investigate verifier agent prompt or add adversarial calibration examples |
| **History poisoning events** | Detect repeated failed strategy in same session or across sessions | 100% of runs that hit max repairs | If detected > 0, fix history read/write logic before next run |

## 7. Production Monitoring

**Tracing Tool:** LangSmith

LangSmith provides:
- Automatic LangGraph trace capture (nodes, edges, state transitions, tool calls)
- Token usage per node and total run
- Checkpoint snapshots at each phase boundary
- Custom span attributes for domain-specific metadata (confidence score, repair count, etc.)

**Setup:**
```bash
# Environment variables
export LANGSMITH_TRACING=true
export LANGSMITH_API_KEY=<your-key>
export LANGSMITH_PROJECT=h2ui-agent-prod

# In code (src/agent/tracing.ts)
import { trace } from "@langchain/langsmith";

export const tracer = trace({
  projectName: "h2ui-agent-prod",
  metadata: {
    agent_version: "1.0.0",
    framework: "langgraph",
  },
});
```

**Key Metrics to Track:**

| Metric | What It Measures | LangSmith Query |
|--------|------------------|-----------------|
| **Avg confidence score** | Overall output quality across all runs | `aggregate(avg(trace.metadata.confidence_score))` |
| **Repair attempt distribution** | How often agent needs repairs | `aggregate(count(trace where repair_attempts > 0)) / aggregate(count(trace))` |
| **Token per conversion** | Cost efficiency | `aggregate(avg(total_tokens))` |
| **Phase duration (p95)** | Latency breakdown per phase | `aggregate(percentile(phase_duration_ms, 95))` grouped by phase |
| **Error rate** | Tool failures, schema validation failures | `aggregate(count(trace where error is not null))` |

**Alert Thresholds:**

| Alert | Threshold | Severity | Action |
|-------|-----------|----------|--------|
| **Confidence degradation** | Avg confidence drops below 70% over 1 hour | Warning | Page on-call; investigate recent changes |
| **Repair loop storm** | > 30% of runs hit max repairs in 15 min | Critical | Page on-call; disable agent mode via feature flag |
| **Token budget breach** | Any single run exceeds 50k tokens | Warning | Log for analysis; agent already self-limited |
| **Tool call failure rate** | > 5% of tool calls fail validation in 5 min | Warning | Page on-call; check schema definitions |
| **Phase latency spike** | p95 latency > 2x baseline for PLAN or VERIFY phase | Warning | Page on-call; check LLM API status |

**Smart Sampling Strategy:**

Select interactions for human review using signal-based filters, not random sampling:

1. **High-priority samples** (always review):
   - Runs with confidence < 70%
   - Runs that exhausted all 3 repair attempts
   - Runs with token count > 30k (high cost)
   - Runs flagged by online guardrail (silent corruption detection)

2. **Random samples** (10 per day):
   - 10 randomly selected runs from the day's traffic, regardless of outcome
   - Ensures we catch failure modes not covered by signal-based filters

3. **Regression samples** (post-deploy only):
   - All runs that differ from previous deployment (diff-based)
   - Catches regressions that don't trigger alert thresholds but degrade quality

**Sampling Pipeline:**
```typescript
// Pseudocode for sampling decision
function shouldSample(trace: Trace): boolean {
  // Always sample low confidence
  if (trace.confidence_score < 0.70) return true;

  // Always sample exhausted repairs
  if (trace.repair_attempts >= 3) return true;

  // Always sample high cost
  if (trace.total_tokens > 30000) return true;

  // Always sample guardrail flags
  if (trace.guardrail_flags.length > 0) return true;

  // Random 2% of remaining
  if (Math.random() < 0.02) return true;

  return false;
}
```
